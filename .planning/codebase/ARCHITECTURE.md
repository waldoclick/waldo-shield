# Architecture

**Analysis Date:** 2026-03-16

## Pattern Overview

**Overall:** Plugin-based CLI scanner with a modular analysis pipeline

**Key Characteristics:**
- Single entry-point CLI (`src/scanner.py`) orchestrates all analysis
- Each security concern is isolated in its own module under `src/modules/`
- All modules expose a single `analyze(url: str) -> dict` interface contract
- Results are aggregated into a unified JSON report with severity-based scoring
- Parallel execution for network-independent modules (DNS, port scan); sequential for HTTP-dependent modules

## Layers

**Orchestration Layer:**
- Purpose: Parse CLI arguments, dispatch modules, aggregate results, calculate risk score, write output
- Location: `src/scanner.py`
- Contains: `scan()`, `run_module()`, `calculate_risk_score()`, `print_summary()`, `main()`
- Depends on: All modules in `src/modules/`
- Used by: CLI invocation (`python src/scanner.py <url>`)

**Analysis Modules Layer:**
- Purpose: Perform a specific security check against a URL and return structured findings
- Location: `src/modules/`
- Contains: Six independent modules, each implementing `analyze(url) -> dict`
- Depends on: External Python libraries (requests, dnspython, ssl, socket, beautifulsoup4)
- Used by: `src/scanner.py` via `run_module()`

**Output Layer:**
- Purpose: Persist scan results as JSON reports for tracking remediation over time
- Location: `reports/` directory
- Contains: JSON files named `<hostname>_report_<timestamp>.json`
- Generated by: `src/scanner.py` after aggregating all module results

**Session/Documentation Layer:**
- Purpose: Human-authored progress logs tracking fixes applied and pending work
- Location: `sessions/YYYY-MM-DD.md`
- Contains: Completed fixes, false positives, risk score deltas, pending actions
- Used by: Human operators to understand current security state

## Data Flow

**Full Scan Pipeline:**

1. `main()` in `src/scanner.py` parses CLI args (`url`, `--modules`, `--output`)
2. `normalize_url()` ensures the URL has an `https://` or `http://` scheme
3. `scan()` builds a `module_map` dict mapping module keys to `(name, analyze_func)` tuples
4. DNS and port modules run in parallel via `concurrent.futures.ThreadPoolExecutor(max_workers=2)`
5. HTTP-dependent modules (`headers`, `ssl`, `tech`, `vulns`) run sequentially
6. Each module is wrapped in `run_module()` which captures timing and catches exceptions
7. Each module returns a dict with at minimum `{"module": str, "issues": list, ...module-specific fields}`
8. All `issues` lists are collected and flattened into `all_issues` with a `source_module` field added
9. Issues are sorted by `SEVERITY_ORDER` (critical → high → medium → low → info)
10. `calculate_risk_score()` computes a 0–100 score (capped) using `SEVERITY_SCORES` weights
11. Final report dict is assembled with `meta`, `risk_summary`, `all_issues`, and per-module results
12. Report is written to disk as JSON; `print_summary()` outputs human-readable summary to stdout

**Module Result Contract:**

Every module returns a dict conforming to:
```python
{
    "module": str,          # module name identifier
    "url": str,             # scanned URL
    "issues": list[dict],   # list of issue dicts with severity/message/recommendation
    "error": str | None,    # error message if module failed
    # ... module-specific fields
}
```

Each issue dict has:
```python
{
    "severity": "critical" | "high" | "medium" | "low" | "info",
    "message": str,
    "recommendation": str,
    # optional: "header", "url", "source_module" (added by orchestrator)
}
```

**Severity Scoring:**
- `critical` = 10 pts, `high` = 7 pts, `medium` = 4 pts, `low` = 1 pt, `info` = 0 pts
- Score = sum of all issue points, capped at 100
- Risk levels: `none` (0) → `low` (1–19) → `medium` (20–39) → `high` (40–69) → `critical` (≥70)

## Key Abstractions

**Module Interface:**
- Purpose: Uniform contract allowing orchestrator to call any module identically
- Examples: `src/modules/headers.py`, `src/modules/ssl_tls.py`, `src/modules/dns_analysis.py`, `src/modules/port_scan.py`, `src/modules/tech_detection.py`, `src/modules/vulnerabilities.py`
- Pattern: Each module exports one public function `analyze(url: str) -> dict`; all private helpers are prefixed with `_`

**Issue Dict:**
- Purpose: Standardized finding format enabling severity sorting and score aggregation
- Pattern: `{"severity": str, "message": str, "recommendation": str}` — used uniformly across all six modules

**Zero Trust Detection:**
- Purpose: Avoid false positives for Cloudflare Zero Trust-protected routes (302 redirects to `cloudflareaccess.com`)
- Files: `src/modules/vulnerabilities.py` (`_is_zero_trust_redirect()`), `src/modules/tech_detection.py` (inline check)
- Pattern: Check redirect `location` header for `ZERO_TRUST_INDICATORS = ["cloudflareaccess.com", "cdn-cgi/access", "CF_AppSession"]`

**Content Signature Verification:**
- Purpose: Confirm a 200 response for sensitive files (`.env`, `.git/HEAD`, `phpinfo.php`) contains actual sensitive content, not a CMS 404
- Files: `src/modules/vulnerabilities.py` (`CONTENT_SIGNATURES`, `_is_real_sensitive_content()`)
- Pattern: Check first 2KB of response body for known content signatures before raising an issue

## Entry Points

**CLI Entry Point:**
- Location: `src/scanner.py` — `main()` function, line 178
- Triggers: `python src/scanner.py <url> [--output file] [--modules list]`
- Responsibilities: Argument parsing, module selection, output file naming, report serialization

**Module Entry Points:**
- Location: `src/modules/<module>.py` — `analyze(url)` function in each file
- Triggers: Called by `run_module()` in `src/scanner.py`
- Responsibilities: Perform all checks for a specific security domain; return structured dict with issues

## Error Handling

**Strategy:** Catch-and-continue — module errors are captured as `error` fields in the result dict, never allowed to abort the overall scan

**Patterns:**
- `run_module()` in `src/scanner.py` wraps each `analyze()` call in a `try/except Exception` — on failure returns `{"module": name, "error": str(e), "issues": []}`
- Each module internally uses specific exception types (e.g. `ssl.SSLCertVerificationError`, `requests.exceptions.ConnectionError`) with a final `except Exception` fallback
- Sub-check helpers inside modules (e.g. `_test_sensitive_files`, `_check_port`) use bare `except Exception: pass` to silently skip individual probe failures without aborting the module

## Cross-Cutting Concerns

**Logging:** `print()` to stdout only — `run_module()` prints `[*] Running: {name}...` and `[+] Done: {name} ({elapsed}s) - N issue(s) found` with `flush=True`

**Validation:** URL normalization in `normalize_url()` (`src/scanner.py` line 31); module selection validation in `main()` against `all_modules` list

**Authentication:** Not applicable — scanner is a passive external tool; no auth state managed by the scanner itself (target auth handled by Cloudflare Zero Trust, transparent to HTTP modules)

**Concurrency:** `concurrent.futures.ThreadPoolExecutor` used in two places:
- `src/scanner.py`: `max_workers=2` for parallel DNS + port scan
- `src/modules/port_scan.py`: `max_workers=50` for parallel port probing across `COMMON_PORTS`
